### Topic Recap: Pipeline Hazards (Structural, Data, Control)

**Hazard Types**:
- **Structural Hazard**: Hardware resource conflict (e.g., single memory for instruction and data).
- **Data Hazard**: Instruction depends on result of previous instruction (RAW, WAR, WAW).
- **Control Hazard**: Branch or jump changes the instruction flow.

**Solutions**:
- Structural: Add more hardware (separate instruction/data cache).
- Data: Forwarding (bypassing), stalling (inserting bubbles).
- Control: Branch prediction, delayed branching, branch target buffer.

### Problem 1: Structural Hazard

In a 5-stage pipeline with a single unified cache for instructions and data, identify structural hazards.

```
LW R1, 0(R2)    ; Load word (memory access in MEM stage)
ADD R3, R4, R5  ; ALU operation (no MEM stage conflict)
```

In cycle when LW is in MEM stage and next instruction is in IF stage:
- LW accesses memory (MEM)
- Next instruction wants to fetch from memory (IF)
- Both need memory -> structural hazard
- Pipeline stalls for one cycle

**Solution**: Use separate instruction cache and data cache (Harvard architecture).

### Problem 2: Data Hazard Classification

Classify the hazards in this code as RAW, WAR, or WAW.

```
I1: ADD R1, R2, R3   ; R1 = R2 + R3
I2: SUB R2, R1, R4   ; R2 = R1 - R4
I3: ADD R1, R5, R6   ; R1 = R5 + R6
I4: MUL R7, R2, R8   ; R7 = R2 * R8
```

**Solution**:

- I1 -> I2: R1 is written by I1, read by I2 -> RAW (true dependency)
- I1 -> I3: R1 is written by I1, written by I3 -> WAW (output dependency)
- I2 -> I4: R2 is written by I2, read by I4 -> RAW (true dependency)
- I2 -> I3: No dependency (different registers)

RAW hazards are true dependencies that must be resolved (by forwarding or stalling).
WAW hazards only matter if an instruction after I3 reads R1 and expects I3's value.

### Problem 3: Control Hazard Example

A branch instruction is followed by 3 instructions. If the branch is taken, those 3 instructions should not execute. Calculate the penalty with and without branch prediction.

**Solution**:

Without branch prediction:
- Always assume not taken. If branch is taken, flush 3 instructions (3 cycle penalty).
- Or always stall until branch resolves: 3 cycles penalty for every branch.

With perfect branch prediction:
- 0 cycle penalty for correctly predicted branches.
- 3 cycle penalty for mispredicted branches.

Example: 20% branches, 90% prediction accuracy:
- Correct predictions: 0.20 x 0.90 = 18% of instructions, 0 penalty
- Mispredictions: 0.20 x 0.10 = 2% of instructions, 3 cycles penalty
- Average penalty per instruction = 0.02 x 3 = 0.06 cycles
- Effective CPI = 1 + 0.06 = 1.06

Without branch prediction (always stall):
- Average penalty = 0.20 x 3 = 0.60 cycles
- Effective CPI = 1 + 0.60 = 1.60

Branch prediction improves CPI from 1.60 to 1.06.

### Problem 4: Branch Prediction Accuracy

A program has 1000 instructions, 250 of which are branches. A predictor correctly predicts 210 branches and mispredicts 40. Calculate prediction accuracy and the penalty.

**Solution**:

Total branches = 250
Correct predictions = 210
Mispredictions = 40

Accuracy = 210 / 250 = 84%

With 3-cycle misprediction penalty:
- Total stall cycles due to mispredictions = 40 x 3 = 120 cycles
- Ideal execution time = 1000 cycles (CPI = 1)
- Actual execution time = 1000 + 120 = 1120 cycles
- Effective CPI = 1120 / 1000 = 1.12

### Problem 5: Delayed Branch Slot

In a 5-stage pipeline with one delayed branch slot, rearrange this code to fill the slot:

```
Loop:   ADD R1, R2, R3
        SUB R4, R5, R6
        BEQ R1, R0, Loop   ; branch if R1 == 0
        LW R7, 0(R8)
```

**Solution**:

The delayed branch slot is the instruction immediately after the branch. This instruction always executes, regardless of whether the branch is taken or not.

We can move the SUB instruction into the branch delay slot:

```
Loop:   ADD R1, R2, R3
        BEQ R1, R0, Loop   ; branch if R1 == 0
        SUB R4, R5, R6     ; delay slot (always executes)
        LW R7, 0(R8)
```

The SUB instruction does not depend on the branch condition and can safely execute regardless of whether the branch is taken.
