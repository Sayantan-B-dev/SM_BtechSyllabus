### Topic Recap: Cache Coherency

**Cache Coherence Protocols**:
- **MSI**: Modified, Shared, Invalid
- **MESI**: Modified, Exclusive, Shared, Invalid (adds Exclusive state for efficiency)
- **MOESI**: Modified, Owner, Exclusive, Shared, Invalid

**Snooping vs Directory**:
- Snooping: All caches monitor the bus. Suitable for small-scale systems.
- Directory: Centralized/ditributed directory tracks cache states. Scalable.

### Problem 1: MESI Protocol State Transition

Track cache block X through MESI states:

Initial state: X not in any cache.

| Event | P1 State | P2 State |
|-------|----------|----------|
| P1 reads X | Exclusive (E) | - |
| P2 reads X | Shared (S) | Shared (S) |
| P1 writes X | Modified (M) | Invalid (I) |
| P2 reads X | Shared (S) | Shared (S) |

**Explanation**:
- First read: Exclusive (only P1 has it, clean)
- Second read: Shared (multiple caches have it, clean)
- Write: Modified (only P1 has it, dirty). P2's copy invalidated.
- Read by P2: P1 supplies the modified data. Both become Shared (clean).

### Problem 2: Bus Traffic Analysis

2 processors, each with 64 KB cache. A program accesses a shared variable X. P1 writes X every 10 cycles. P2 reads X every 5 cycles. How many bus transactions per 100 cycles?

**Solution**:

P1 writes every 10 cycles: 100/10 = 10 writes
P2 reads every 5 cycles: 100/5 = 20 reads

However, not every read causes a bus transaction. After P1 writes, P2's copy is invalidated, so the next P2 read misses.

Timeline (simplified):
- Cycle 0: P1 writes -> bus transaction (write invalidate)
- Cycle 1: P2 reads -> cache miss -> bus transaction (read)
- Cycle 5: P2 reads -> hit (still valid)
- Cycle 10: P1 writes -> bus transaction (write invalidate)
- Cycle 11: P2 reads -> miss -> bus transaction
- Cycle 15: P2 reads -> hit
- ...

Each P1 write causes the next P2 read to miss. In steady state:
- 10 P1 writes = 10 invalidation bus transactions
- 10 P2 misses (after each P1 write) = 10 read bus transactions
- Total: 20 bus transactions per 100 cycles

### Problem 3: Directory Protocol

In a directory-based system with 4 processors, the directory entry for block X has a bit vector [P1, P2, P3, P4]. Describe the protocol for:

(a) P1 reads X (initially in memory, not cached)
(b) P2 reads X
(c) P1 writes X
(d) P1 writes X again

**Solution**:

(a) P1 reads X:
- Message from P1 to directory: "Read miss X"
- Directory updates vector: [1, 0, 0, 0]
- Directory asks memory to supply X to P1
- P1 caches X in Shared state

(b) P2 reads X:
- Message from P2 to directory: "Read miss X"
- Directory updates vector: [1, 1, 0, 0]
- Directory asks memory to supply X to P2 (or P1 can supply)
- P2 caches X in Shared state

(c) P1 writes X:
- Message from P1 to directory: "Write miss X"
- Directory updates vector: [1, 0, 0, 0] (others invalidated)
- Directory sends invalidation to P2
- P2 invalidates X and sends acknowledgment
- Directory confirms write to P1
- P1 caches X in Modified state

(d) P1 writes X again:
- P1 already has X in Modified state
- No coherence messages needed (P1 has exclusive ownership)
- This is a cache hit

### Problem 4: Coherence Miss vs Capacity Miss vs Compulsory Miss

Classify these cache misses:

| Miss | Type | Reason |
|------|------|--------|
| First access to block X | Compulsory | Never accessed before |
| P2 reads X after P1 wrote to X | Coherence | Invalidated by another processor |
| Program accesses data larger than cache | Capacity | Cache too small for working set |
| Block Y evicted due to mapping conflict | Conflict | Two blocks map to same cache line |
| P3 reads X after P1 and P2 shared it | Coherence (may be Shared hit instead) | Depends on protocol |

### Problem 5: Synchronization with Locks

Describe the cache coherence traffic for a test-and-set lock:

```c
while (test_and_set(&lock, 1) == 1) ;  // spin
// critical section
lock = 0;  // release
```

With 4 processors spinning on the same lock.

**Solution**:

Initially: lock = 0 in memory. All 4 processors start spinning.

1. P1 succeeds in test_and_set (reads lock=0, writes lock=1 atomically)
   - P1: lock in Modified state
   - Other processors: Invalid

2. Other processors spin: test_and_set fails (read misses, get lock=1)
   - Each gets Shared copy of lock=1
   - All keep spinning, reading their cached copy (hits)

3. P1 releases: lock = 0 (write)
   - P1 writes 0, all other copies invalidated

4. All other processors miss, try test_and_set:
   - One succeeds (gets lock=1, writes)
   - Others read lock=1 again (Shared)

The coherence traffic is bursty: one burst on acquire, one burst on release. This is why spin locks are efficient for short critical sections on cache-coherent systems.
