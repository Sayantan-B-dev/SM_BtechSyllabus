### Topic Recap: Parallel Processors, Concurrent Access to Memory

**Parallel Architectures**:
- **SISD**: Single Instruction, Single Data (standard uniprocessor).
- **SIMD**: Single Instruction, Multiple Data (vector processors, GPUs).
- **MISD**: Multiple Instruction, Single Data (rare).
- **MIMD**: Multiple Instruction, Multiple Data (multicore processors, clusters).

**Memory Access Models**:
- **UMA (Uniform Memory Access)**: All processors share same memory with equal access time.
- **NUMA (Non-Uniform Memory Access)**: Each processor has local memory (faster access) and remote memory (slower).
- **COMA (Cache-Only Memory Architecture)**: Main memory acts as a cache.

### Problem 1: Amdahl's Law

If 80% of a program can be parallelized, what is the maximum speedup with 4 processors? With unlimited processors?

**Solution**:

Amdahl's Law: Speedup = 1 / ((1 - P) + P/N)

Where P = parallel fraction, N = number of processors.

With N = 4:
Speedup = 1 / ((1 - 0.80) + 0.80/4) = 1 / (0.20 + 0.20) = 1 / 0.40 = 2.5x

With unlimited processors (N -> infinity):
Speedup = 1 / (1 - 0.80) = 1 / 0.20 = 5x (maximum possible speedup)

Even with infinite processors, the 20% sequential portion limits the speedup to 5x.

### Problem 2: Gustafson's Law

Gustafson's Law argues that problem size scales with the number of processors. If 80% of a program is parallelizable, what speedup is achieved with 4 processors under Gustafson's Law?

**Solution**:

Gustafson's Law: Speedup = N - (N - 1) x S

Where S = sequential fraction, N = number of processors.

With N = 4, S = 0.20:
Speedup = 4 - (4 - 1) x 0.20 = 4 - 3 x 0.20 = 4 - 0.60 = 3.4x

Amdahl's Law assumes fixed problem size. Gustafson's Law assumes the parallel part scales with N, giving a more optimistic speedup.

### Problem 3: Cache Coherence Problem

Two processors share memory. P1 reads X=5, P2 reads X=5. P1 writes X=10. P2 still has X=5 in its cache. Describe the cache coherence problem.

**Solution**:

This is the classic cache coherence problem. P2's cache has stale data (X=5) while main memory and P1 have X=10.

If P2 reads X again, it gets 5 instead of 10, producing incorrect results.

**Solutions**:
- **Snooping Protocol**: Each cache monitors (snoops) the bus. When P1 writes X, P2 invalidates its copy of X.
- **Directory Protocol**: A directory tracks which caches have copies of each block. On write, the directory sends invalidation messages.

### Problem 4: Snooping Protocol Example

For a write-invalidate snooping protocol, show the state transitions for:

1. P1 reads X
2. P2 reads X
3. P1 writes X
4. P2 reads X

**Solution**:

Initial: X is in main memory, not in any cache.

Step 1: P1 reads X
- Bus request: "Read X"
- Memory supplies X
- P1 cache: X in "Shared" state
- P2 cache: X not present

Step 2: P2 reads X
- Bus request: "Read X"
- P1 snoops: "I have X in Shared, no need to intervene"
- Memory supplies X
- P2 cache: X in "Shared" state

Step 3: P1 writes X
- Bus request: "Read-for-Invalidation X" (or "Write X")
- P1 cache: X transitions to "Modified" state
- P2 snoops: X is invalidated in P2 cache
- P2 cache: X in "Invalid" state

Step 4: P2 reads X
- Bus request: "Read X"
- P1 snoops: "I have Modified copy, I will supply it"
- P1 writes X back to memory (or supplies directly)
- P2 cache: X in "Shared" state
- P1 cache: X in "Shared" state (or stays Modified)

### Problem 5: False Sharing

Two processors each have a variable in the same cache block:
- P1 repeatedly reads/writes A
- P2 repeatedly reads/writes B
- A and B are in the same 64-byte cache block

**Solution**:

Even though P1 and P2 access different variables, they share a cache block. This causes:

1. P1 writes A -> block becomes Modified in P1, Invalid in P2
2. P2 writes B -> block becomes Modified in P2, Invalid in P1 (cache miss for P1)
3. P1 writes A -> block becomes Modified in P1, Invalid in P2 (cache miss for P2)

This ping-pong effect causes excessive cache misses even though there is no true data sharing. This is called **false sharing**.

**Fix**: Place A and B in different cache blocks (add padding).
