### 1. Introduction to Interrupts and Exceptions

**Interrupts** and **exceptions** are mechanisms that cause the processor to temporarily suspend its current execution, save its state, and transfer control to a special handler routine. They are fundamental to multitasking, device I/O, error handling, and system calls.

**Basic distinction:**
- **Interrupts** are **asynchronous** events caused by external devices (hardware interrupts) or software instructions (software interrupts).
- **Exceptions** are **synchronous** events caused by the execution of an instruction itself (e.g., divide by zero, page fault).

---

### 2. Software Interrupts

A **software interrupt** is generated by executing a special instruction in the program (e.g., `INT n` on x86, `SVC` on ARM, `ECALL` on RISC-V). It is used primarily for **system calls** and **debugging**.

#### 2.1 INT Instruction (x86)

The x86 architecture provides the `INT n` instruction, where `n` is a vector number (0-255).

```asm
INT 0x80     ; System call interrupt (Linux)
INT 0x21     ; DOS system call interrupt
INT 0x10     ; Video services (BIOS)
INT 0x13     ; Disk services (BIOS)
```

**What happens when INT n executes:**

```
   Before INT:
      User Mode (Ring 3)
      CS:EIP -> next instruction
      SS:ESP -> user stack
   
   Step 1: CPU reads IDT entry `n` (Interrupt Descriptor Table)
           Checks DPL (Descriptor Privilege Level) against CPL
   
   Step 2: CPU switches to kernel stack (if CPL changes)
           - Loads SS and ESP from TSS (Task State Segment)
   
   Step 3: CPU saves context on kernel stack:
           - SS (user stack segment)
           - ESP (user stack pointer)
           - EFLAGS
           - CS (user code segment)
           - EIP (return address)
   
   Step 4: CPU clears IF (interrupt flag) if interrupt gate, 
           keeps IF if trap gate
   
   Step 5: CPU loads CS:EIP from IDT entry
           - Switches to kernel mode (Ring 0)
   
   Step 6: Handler executes (e.g., system_call entry point)
   
   Step 7: IRET instruction returns:
           - Pops EIP, CS, EFLAGS, ESP, SS
           - Returns to user mode
```

#### 2.2 System Calls via Software Interrupts

On Linux (x86, legacy method):

```asm
; System call: write(1, "Hello", 5)
MOV EAX, 4        ; sys_write syscall number
MOV EBX, 1        ; fd = stdout
MOV ECX, message  ; buf = pointer to string
MOV EDX, 5        ; count = 5
INT 0x80          ; software interrupt -> kernel
```

The kernel's system call handler:
1. Saves all registers on the kernel stack.
2. Validates the system call number (EAX).
3. Validates the arguments (permission checks, address validity).
4. Executes the requested kernel function.
5. Stores the return value in EAX.
6. Restores registers.
7. Executes IRET to return to user mode.

#### 2.3 Software Interrupt vs Hardware Interrupt

| Feature              | Software Interrupt              | Hardware Interrupt             |
|----------------------|---------------------------------|--------------------------------|
| Cause                | INT instruction in code         | External device signal         |
| Timing               | Synchronous (programmed)        | Asynchronous (random)          |
| Predictability       | Fully predictable               | Not predictable                |
| Primary use          | System calls, debugging         | Device I/O, timer              |
| Vector source        | Specified in instruction        | From interrupt controller      |
| Can be disabled?     | No (but OS can block syscall)   | Yes (maskable interrupts)      |

### 3. Exceptions

**Exceptions** are **synchronous** events that occur as a direct result of instruction execution. They are classified into three categories: **faults**, **traps**, and **aborts**.

#### 3.1 Faults

A **fault** is an exception that is detected **during** instruction execution, typically before the instruction completes. The saved return address points to the **faulting instruction**, so the handler can fix the problem and re-execute the instruction.

**Fault Examples:**

| Fault                    | Cause                                    | Typical Handler Action            |
|--------------------------|------------------------------------------|-----------------------------------|
| Divide by zero (#DE)     | DIV or IDIV with divisor = 0             | Send SIGFPE to process            |
| Page fault (#PF)         | Access to a page not in physical memory  | Load page from disk (demand paging)|
| General protection (#GP) | Privileged instruction in user mode, or segment violation | Send SIGSEGV to process |
| Stack fault (#SS)        | Stack segment violation                  | Send SIGSEGV to process            |
| Alignment check (#AC)    | Unaligned memory access (if enabled)     | Send SIGBUS to process             |

**Page Fault Handling (Demand Paging):**

```
   User Program: MOV EAX, [0x12345678]
        |
        v
   CPU tries to translate virtual address 0x12345678
        |
        v
   MMU looks up page table entry (PTE):
        - Present bit = 0 (page not in physical memory)
        |
        v
   CPU triggers Page Fault (#PF) exception
        |
        v
   OS Page Fault Handler:
    1. Determines which virtual address caused the fault
       (read from CR2 register on x86)
    2. Checks if the access is valid (mapped address)
    3. If invalid: SIGSEGV, terminate process
    4. If valid: 
       a. Find a free physical page frame (or evict one)
       b. Start disk I/O to load page from swap/file
       c. Mark process as waiting for I/O
       d. Schedule another process (context switch)
    5. Disk I/O completes:
       a. Update PTE with physical frame address, set Present=1
       b. Mark process as ready to run
    6. Scheduler runs process again
        |
        v
   CPU retries the faulting instruction: MOV EAX, [0x12345678]
        - This time the page is present, so the instruction succeeds.
```

#### 3.2 Traps

A **trap** is an exception that occurs **after** the instruction has completed execution. The saved return address points to the **next instruction** after the one that caused the trap.

**Trap Examples:**

| Trap                      | Cause                                    | Handler Action                    |
|---------------------------|------------------------------------------|-----------------------------------|
| Breakpoint (#BP)          | INT 3 instruction (x86)                  | Debugger breakpoint                |
| Overflow (#OF)            | INTO instruction (x86, obsolete)         | Handle overflow condition          |
| System call (via INT/SVC) | Software interrupt for syscall           | Execute kernel service             |

**Breakpoint Example:**

```
   Program: 
     MOV EAX, 1        ; Normal instruction
     INT 3             ; Breakpoint trap (1-byte instruction: 0xCC)
     ADD EBX, EAX      ; Next instruction (this is saved as return address)
   
   Trap Handler (Debugger):
     - Return address points to ADD instruction
     - Debugger can inspect/modify registers, memory
     - Debugger executes IRET to resume at ADD
```

**Key difference between fault and trap:**

```
   Fault:  | instruction | <- return address (re-execute)
           |             |
   Trap:   | instruction | <- completes
           | next instr  | <- return address (continue)
```

#### 3.3 Aborts

An **abort** is a severe exception indicating a serious hardware or system error. The saved return address may not point to the faulting instruction, and the program typically cannot be recovered. Aborts usually result in system panic or kernel crash dump.

**Abort Examples:**

| Abort                    | Cause                                    | Typical Outcome                |
|--------------------------|------------------------------------------|--------------------------------|
| Double fault (#DF)       | Another exception occurs while handling an exception | System reset / crash dump |
| Machine check (#MC)      | Hardware error detected (ECC memory, bus error) | System halt / panic |
| Shutdown                 | Catastrophic error                       | System halt                    |

**Double Fault Scenario:**

```
   Step 1: DIV instruction divides by zero
        -> #DE (Divide Error) exception
   
   Step 2: CPU tries to push error code, CS, EIP onto stack
        - But stack segment is invalid (e.g., stack pointer corrupted)
        -> #SS (Stack Segment Fault) occurs while handling #DE
   
   Step 3: CPU tries to handle #SS
        - But this second exception cannot be handled either
        -> #DF (Double Fault)
   
   Step 4: CPU loads IDT entry for Double Fault
        - If the double fault handler also fails -> Triple Fault -> System Reset
```

### 4. Exception Handling Mechanism

#### Exception Vector Table / IDT

All interrupts and exceptions are identified by a **vector number** (0-255). The processor uses this number to index into the **Interrupt Descriptor Table** (IDT) on x86, or **Exception Vector Table** on ARM.

```
   x86 IDT (Interrupt Descriptor Table):
   
   Vector 0:  #DE - Divide Error
   Vector 1:  #DB - Debug Exception
   Vector 2:  NMI - Non-Maskable Interrupt
   Vector 3:  #BP - Breakpoint
   Vector 4:  #OF - Overflow
   Vector 5:  #BR - BOUND Range Exceeded
   Vector 6:  #UD - Invalid Opcode
   Vector 7:  #NM - Device Not Available
   Vector 8:  #DF - Double Fault
   Vector 9:  (reserved)
   Vector 10: #TS - Invalid TSS
   Vector 11: #NP - Segment Not Present
   Vector 12: #SS - Stack-Segment Fault
   Vector 13: #GP - General Protection Fault
   Vector 14: #PF - Page Fault
   Vector 15: (reserved)
   Vector 16: #MF - x87 FPU Floating-Point Error
   Vector 17: #AC - Alignment Check
   Vector 18: #MC - Machine Check
   Vector 19: #XM - SIMD Floating-Point Exception
   Vectors 20-31: (reserved)
   Vectors 32-255: Available for OS-defined interrupts (hardware IRQs, system calls)
   
   Each IDT entry (8 bytes on x86):
   +----+----+----+----+----+----+----+----+
   | Offset 15:0 | Segment Selector | Flags | Offset 31:16 |
   +----+----+----+----+----+----+----+----+
```

#### Exception Handling Flow

```
   CPU executes instruction
        |
        v
   Exception detected (synchronous)
        |
        v
   CPU completes current instruction (or aborts mid-execution for faults)
        |
        v
   CPU saves context (SS:ESP, EFLAGS, CS:EIP, optional error code)
        |
        v
   CPU switches to kernel mode (if not already)
        |
        v
   CPU looks up IDT entry by vector number
        |
        v
   CPU loads CS:EIP from IDT entry -> jumps to handler
        |
        v
   Handler executes:
     - Saves additional registers
     - Determines cause and takes action
     - Restores registers
     - Executes IRET
        |
        v
   CPU restores saved context
        |
        v
   Resumes execution at the appropriate address
     - Fault: re-executes the original instruction
     - Trap: executes the next instruction
     - Abort: does not resume (system halt)
```

### 5. Difference Between Interrupt and Exception

| Aspect              | Interrupt                           | Exception                         |
|---------------------|-------------------------------------|-----------------------------------|
| Source              | External device or software INT     | Internal to the executing program |
| Timing              | Asynchronous (can occur at any time)| Synchronous (caused by instruction)|
| Return address      | Points to next instruction          | Fault: points to faulting instruction; Trap: points to next instruction |
| Re-execution        | Not applicable (no instruction faulted) | Faults: instruction is re-executed |
| Predictability      | Unpredictable (random device events)| Predictable (same input -> same exception)|
| Examples            | Timer IRQ, keyboard IRQ, INT 0x80  | Divide by zero, page fault, breakpoint |

## Practice Problems

**Q1:** Explain the difference between a fault and a trap, and give an example of each.

**A1:** A fault occurs during instruction execution and the saved return address points to the faulting instruction so it can be re-executed after the handler fixes the problem. Example: page fault (#PF) -- the fault handler loads the required page from disk, then the instruction that caused the fault is retried. A trap occurs after instruction completion and the saved return address points to the next instruction. Example: breakpoint (#BP via INT 3) -- the trap handler (debugger) inspects state, then resumes at the instruction after the INT 3.

**Q2:** Describe what happens step by step when a user program executes an `INT 0x80` instruction on x86 Linux.

**A2:** (1) CPU pushes SS, ESP, EFLAGS, CS, EIP onto kernel stack (switching stacks if needed). (2) CPU clears IF (if using interrupt gate) or leaves IF set (if using trap gate). (3) CPU loads CS:EIP from IDT entry 0x80, switching to kernel mode (CPL=0). (4) Kernel system_call entry executes: saves all registers on kernel stack. (5) Validates syscall number in EAX using sys_call_table. (6) For valid calls: checks arguments, performs operation. (7) Stores return value in EAX. (8) Restores registers. (9) Executes IRET: pops EIP, CS, EFLAGS, ESP, SS, returns to user mode at the instruction after INT 0x80.

**Q3:** What is a double fault and what typically causes it?

**A3:** A double fault (#DF, vector 8) occurs when an exception is triggered while the CPU is already trying to handle a previous exception. For example, if a divide-by-zero exception occurs and the CPU attempts to push the error code and return address onto the stack, but the stack segment is invalid (causing a stack fault), the CPU cannot handle two exceptions sequentially and raises a double fault. If the double fault handler also fails, a triple fault occurs, causing the system to reset.

**Q4:** Distinguish between hardware interrupts and software interrupts.

**A4:** Hardware interrupts are generated by external devices via the interrupt controller (PIC/APIC). They are asynchronous and unpredictable. Software interrupts are generated by executing the INT n instruction in code. They are synchronous and predictable. Hardware interrupts handle device I/O; software interrupts are used for system calls and debugging. Hardware interrupts can be masked (except NMI); software interrupts always execute (though the OS can choose to deny the requested service).

**Q5:** Why is the page fault exception considered a fault and not a trap?

**A5:** A page fault is a fault because the instruction that accessed the missing page has NOT completed execution. The saved return address (CS:EIP) points to the faulting instruction itself, not the next instruction. After the OS handles the fault (loads the page from disk into physical memory), the CPU retries the original instruction. This time the page is present and the instruction completes successfully. If page faults were traps, the return address would skip past the faulting instruction, and that instruction would never be re-executed, causing the memory access to be lost.
