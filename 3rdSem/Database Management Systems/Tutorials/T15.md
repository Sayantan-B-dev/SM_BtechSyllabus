### Topic Recap: Data Warehousing, OLAP vs OLTP, Star/Snowflake Schema, Data Mining

**Data Warehouse**: Central repository for historical data from multiple sources. Optimized for analysis (read-heavy).

**OLTP vs OLAP**:

| Feature | OLTP | OLAP |
|---------|------|------|
| Purpose | Day-to-day transactions | Analysis and reporting |
| Users | Clerks, customers | Analysts, managers |
| Data | Current, detailed | Historical, summarized |
| Queries | Simple, short | Complex, aggregation |
| Design | Normalized (3NF) | Denormalized (star/snowflake) |
| Size | GB | TB to PB |
| Example | Order processing | Sales trend analysis |

**Star Schema**: Fact table (measures) + Dimension tables (attributes). Central star layout.

**Snowflake Schema**: Dimension tables further normalized into sub-dimensions.

**Data Mining**: Discovering patterns in large datasets. Techniques: classification, clustering, association, regression.

### Problem 1: Star Schema Design

Design a star schema for a retail sales data warehouse.

**Solution**:

**Fact Table**: Sales_Fact
- SaleID (PK)
- DateKey (FK -> Date_Dim)
- ProductKey (FK -> Product_Dim)
- CustomerKey (FK -> Customer_Dim)
- StoreKey (FK -> Store_Dim)
- QuantitySold
- UnitPrice
- TotalAmount

**Dimension Tables**:
- Date_Dim(DateKey, Date, Day, Month, Quarter, Year, Weekday)
- Product_Dim(ProductKey, ProductID, ProductName, Category, Brand, Supplier)
- Customer_Dim(CustomerKey, CustomerID, Name, City, State, Segment)
- Store_Dim(StoreKey, StoreID, StoreName, City, Region, Manager)

This star schema allows queries like:
- Total sales by year, quarter, month
- Top-selling products by category
- Sales by region and customer segment

### Problem 2: OLAP Operations

Given a sales cube with dimensions (Time, Product, Location) and measure (Sales):

- **Slice**: Fix one dimension. E.g., Sales for 2026 (Time=2026).
- **Dice**: Select sub-cube. E.g., Sales for 2026, Electronics, USA.
- **Roll-up**: Aggregate. E.g., Sales by Year (from month-level).
- **Drill-down**: De-aggregate. E.g., Sales by Day (from month-level).
- **Pivot**: Reorient cube. E.g., Swap Time and Product axes.
- **Drill-across**: Combine with another cube.

Example SQL for roll-up:
```sql
SELECT Year, SUM(TotalAmount) AS AnnualSales
FROM Sales_Fact f
JOIN Date_Dim d ON f.DateKey = d.DateKey
GROUP BY Year;
```

### Problem 3: Snowflake Schema

Convert the star schema Product dimension to snowflake.

**Star**: Product_Dim(ProductKey, ProductID, ProductName, CategoryID, CategoryName, BrandID, BrandName)

**Snowflake**:
- Product_Dim(ProductKey, ProductID, ProductName, CategoryID, BrandID)
- Category_Dim(CategoryID, CategoryName, CategoryDescription)
- Brand_Dim(BrandID, BrandName, BrandWebsite)

**Trade-off**: Snowflake saves storage (no redundancy) but requires more joins for queries. Star is simpler and faster for querying.

### Problem 4: Association Rule Mining

Given transaction data:

| Transaction | Items |
|-------------|-------|
| T1 | Bread, Milk, Eggs |
| T2 | Bread, Butter, Milk |
| T3 | Bread, Eggs |
| T4 | Milk, Eggs, Butter |
| T5 | Bread, Milk, Butter, Eggs |

Find association rules with minimum support 60% and minimum confidence 80%.

**Solution**:

Support(X) = Transactions containing X / Total transactions

Support(Bread) = 4/5 = 80%
Support(Milk) = 4/5 = 80%
Support(Eggs) = 4/5 = 80%
Support(Butter) = 3/5 = 60%

Support(Bread, Milk) = 3/5 = 60%
Support(Bread, Eggs) = 3/5 = 60%
Support(Milk, Eggs) = 3/5 = 60%

Confidence(Bread -> Milk) = Support(Bread, Milk) / Support(Bread) = 60%/80% = 75%
Confidence(Milk -> Bread) = 60%/80% = 75%
Confidence(Bread -> Eggs) = 60%/80% = 75%
Confidence(Milk -> Eggs) = 60%/80% = 75%
Confidence(Eggs -> Milk) = 60%/80% = 75%

None reach 80% confidence. Try 3-item sets:

Support(Bread, Milk, Eggs) = 2/5 = 40% (below 60%)

No strong association rules at these thresholds. Lower threshold to 70%:
Confidence(Milk -> Eggs) = 75% >= 70%. Rule: "If Milk, then Eggs" (75% confidence).

### Problem 5: ETL Process

Describe the ETL (Extract, Transform, Load) process for a data warehouse.

**Extract**:
- Read data from source systems (OLTP databases, CSV files, APIs).
- Handle different formats and encodings.
- Incremental vs full extraction.

**Transform**:
- Clean data (handle missing values, duplicates).
- Convert data types and formats.
- Apply business rules (e.g., standardize currency).
- Generate surrogate keys.
- Aggregate data if needed.

**Load**:
- Insert into dimension tables first (generate keys).
- Insert into fact tables (using generated keys).
- Update indexes and statistics.
- Verify data quality (row counts, sum checks).
