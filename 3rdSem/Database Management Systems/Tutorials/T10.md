### Topic Recap: Transactions, ACID Properties, Serializability

**Transaction**: A logical unit of work (one or more SQL operations) that transforms the database from one consistent state to another.

**ACID Properties**:
- **Atomicity**: All or nothing.
- **Consistency**: Transaction brings DB from one valid state to another.
- **Isolation**: Concurrent transactions appear to run sequentially.
- **Durability**: Committed changes persist.

**Serializability**: Schedule equivalent to some serial execution.
- **Conflict Serializability**: Swapping non-conflicting operations.
- **View Serializability**: Same read/write sets as some serial schedule.

### Problem 1: ACID Example

Transaction T: Transfer $100 from Account A to Account B.

```
T: READ(A)
   A = A - 100
   WRITE(A)
   READ(B)
   B = B + 100
   WRITE(B)
```

How does each ACID property apply?

**Atomicity**: Both debit and credit must complete. If system crashes after WRITE(A) but before WRITE(B), $100 is lost. Atomicity requires rollback.

**Consistency**: Sum A+B must remain unchanged. Before: A+B = X. After: (A-100)+(B+100) = X. Consistent.

**Isolation**: If another transaction reads A and B between the debit and credit, it sees an inconsistent state. Isolation requires that intermediate states are invisible.

**Durability**: Once committed, the $100 transfer persists even after a power failure.

### Problem 2: Conflict Serializability

Check if this schedule is conflict-serializable:

```
T1: R(A), W(A), R(B), W(B)
T2: R(A), W(A), R(C), W(C)
```

Schedule S:
```
T1: R(A) -> T1: W(A) -> T2: R(A) -> T2: W(A) -> T1: R(B) -> T1: W(B) -> T2: R(C) -> T2: W(C)
```

**Solution**:

Conflicting operations (same data item, at least one write):
- T1:W(A) and T2:R(A) -> conflict (T1 before T2)
- T1:W(A) and T2:W(A) -> conflict (T1 before T2)
- T2:R(A) and T1:W(A) -> already counted
- T2:W(A) and T1:W(A) -> already counted

Precedence graph:
- T1 -> T2 (because T1:W(A) before T2:R(A) and T2:W(A))

No cycle. Schedule is conflict-serializable (equivalent to T1;T2).

### Problem 3: Non-Serializable Schedule

Show that this schedule is NOT serializable:

```
T1: R(A) -> T2: R(A) -> T1: W(A) -> T2: W(A) -> T1: W(B) -> T2: R(B)
```

**Solution**:

Conflicts:
- T1:W(A) and T2:W(A) -> T1 before T2
- T2:W(A) and T1:W(A) -> T2 before T1 (wait, same conflict, we need to be careful)

Let me use a clearer example:

```
T1: R(A) -> T2: W(A) -> T1: W(A) -> T2: R(B)
```

Conflicts:
- T2:W(A) and T1:W(A) -> T2 -> T1
- T1:W(A) and T2: ... no subsequent T2 operation on A

Hmm, let me use a classic non-serializable schedule:

```
T1: R(A) -> T2: R(B) -> T1: W(A) -> T2: W(B) -> T1: R(B) -> T2: W(A)
```

Wait, a simpler example that produces a cycle:

```
S: T1:W(A) -> T2:W(A) -> T2:W(B) -> T1:W(B)
```

Conflicts:
- T1:W(A) and T2:W(A) -> T1 -> T2
- T2:W(B) and T1:W(B) -> T2 -> T1

Precedence: T1 -> T2 -> T1 (CYCLE!). Not conflict-serializable.

### Problem 4: Recoverable vs Cascadeless Schedules

Given schedule:
```
T1: W(A) -> T2: R(A) -> T1: COMMIT -> T2: COMMIT
```

**Recoverable?** Yes. T2 reads A written by T1. T2 commits after T1 commits. If T1 aborts, T2 can also be aborted (cascading abort possible).

**Cascadeless?** No. T2 reads A before T1 commits. If T1 aborts, T2 must also abort.

Cascadeless schedule version:
```
T1: W(A) -> T1: COMMIT -> T2: R(A) -> T2: COMMIT
```
Now T2 reads A only after T1 commits. No cascading abort possible.

### Problem 5: View Serializability

Schedule S:
```
T1: R(A) -> T2: W(A) -> T1: W(A)
```

Is this view-serializable?

**Solution**:

Check blind writes: T2:W(A) is a blind write (T2 doesn't read A first).

Check if view-equivalent to a serial schedule:
- Both T1;T2 and T2;T1 are possible serial schedules.
- In T1;T2: T1 reads A, writes A, then T2 writes A.
- In T2;T1: T2 writes A, then T1 reads A, writes A.
- In S: Same as T2;T1 (T1 reads the initial A, T2 writes A, T1 writes over it).

S is view-equivalent to T2;T1. View-serializable.
