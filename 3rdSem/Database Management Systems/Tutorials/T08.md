## Notes

### Topic Recap: Normalization (1NF to 6NF), Dependency Preservation

**Normal Forms**:
- **1NF**: Atomic values, no repeating groups.
- **2NF**: 1NF + every non-key attribute fully functionally dependent on the whole primary key.
- **3NF**: 2NF + no transitive dependency (non-key -> non-key).
- **BCNF**: 3NF + every determinant is a candidate key.
- **4NF**: BCNF + no multi-valued dependencies.
- **5NF**: 4NF + every join dependency implied by candidate keys.
- **6NF**: No non-trivial join dependencies (temporal databases).

**Functional Dependency (FD)**: X -> Y means X determines Y.

### Problem 1: Identify 1NF Violation

Table: StudentCourses(RollNo, Name, Courses)

| RollNo | Name | Courses |
|--------|------|---------|
| 101 | Alice | CS301, CS302 |
| 102 | Bob | EE201 |

**Problem**: Courses column has multiple values (repeating group).

**Fix (1NF)**: Convert to atomic values.

| RollNo | Name | Course |
|--------|------|--------|
| 101 | Alice | CS301 |
| 101 | Alice | CS302 |
| 102 | Bob | EE201 |

### Problem 2: 2NF Decomposition

Table: Enrollment(RollNo, CourseCode, StudentName, Instructor, Grade)

Given FDs: RollNo -> StudentName, CourseCode -> Instructor, (RollNo, CourseCode) -> Grade

**Problem**: Partial dependencies — StudentName depends only on RollNo (part of PK), Instructor depends only on CourseCode (part of PK).

**Fix (2NF)**: Decompose into three tables:

```
Student(RollNo, StudentName)  -- FD: RollNo -> StudentName
Course(CourseCode, Instructor)  -- FD: CourseCode -> Instructor
Enrollment(RollNo, CourseCode, Grade)  -- FD: (RollNo, CourseCode) -> Grade
```

### Problem 3: 3NF Decomposition

Table: Employee(EID, Name, DeptID, DeptLocation, Manager)

FDs: EID -> Name, DeptID; DeptID -> DeptLocation, Manager; EID -> DeptLocation (transitive)

**Problem**: Transitive dependency: EID -> DeptID -> DeptLocation (non-key -> non-key).

**Fix (3NF)**: Decompose:

```
Employee(EID, Name, DeptID)  -- FDs: EID -> Name, DeptID
Department(DeptID, DeptLocation, Manager)  -- FDs: DeptID -> DeptLocation, Manager
```

Now there are no transitive dependencies.

### Problem 4: BCNF Decomposition

Table: StudentSubject(RollNo, SubjectCode, Professor, Grade)

FDs: (RollNo, SubjectCode) -> Grade, Professor
     Professor -> SubjectCode

**Problem**: Professor -> SubjectCode, but Professor is not a candidate key (the PK is RollNo, SubjectCode).

**Fix (BCNF)**: Decompose:

```
SubjectProfessor(Professor, SubjectCode)  -- FD: Professor -> SubjectCode
Enrollment(RollNo, Professor, Grade)  -- No non-trivial FDs other than candidate key
```

Or equivalently:

```
SubjectProfessor(Professor, SubjectCode)
Enrollment(RollNo, SubjectCode, Grade)
```

### Problem 5: Dependency Preservation

Check if the BCNF decomposition from Problem 4 preserves all FDs.

Original FDs: F = { (RollNo, SubjectCode) -> Grade, Professor, Professor -> SubjectCode }

Decomposition: R1(Professor, SubjectCode), R2(RollNo, Professor, Grade)

**Check**:
- FD1: Professor -> SubjectCode. All attributes in R1. Preserved.
- FD2: (RollNo, SubjectCode) -> Grade. Attributes across R1 and R2. Not directly in any relation. But can we derive it? In R2 we have (RollNo, Professor) -> Grade. Combined with R1's Professor -> SubjectCode, we can derive (RollNo, SubjectCode) -> Grade.

The decomposition is dependency-preserving because the closure of the union of FDs in R1 and R2 equals the closure of F.
