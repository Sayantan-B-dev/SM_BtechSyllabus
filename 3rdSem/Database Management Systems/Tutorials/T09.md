## Notes

### Topic Recap: Indexing Techniques, B-Trees, B+ Trees, Hashing

**Indexing**:
- **Ordered Index**: Primary (clustering) and secondary (non-clustering) indexes.
- **B-Tree**: Balanced tree. Internal nodes and leaf nodes contain keys and pointers.
- **B+ Tree**: Only leaves contain data pointers. Internal nodes are routing nodes. All leaves at same depth, linked for sequential access.
- **Hashing**: Static (bucket) and dynamic (extendible, linear) hashing.

### Problem 1: Dense vs Sparse Index

Given 1000 records sorted by RollNo, compare dense and sparse indexes.

**Dense Index**: One index entry per record. 1000 entries. Faster lookup (direct pointer to record).

**Sparse Index**: One index entry per block. If each block holds 10 records, 100 entries. Slower (need to scan block after locating it).

| Criteria | Dense | Sparse |
|----------|-------|--------|
| Size | Larger | Smaller |
| Lookup speed | Faster (1 step) | Slower (2 steps: index + block scan) |
| Update cost | Higher (update every entry) | Lower (update only when block changes) |
| Storage overhead | More | Less |

### Problem 2: B+ Tree Search

A B+ tree of order 4 (max 4 keys per node, min 2 keys) contains keys: 10, 20, 30, 40, 50, 60, 70, 80, 90.

Trace search for key 45.

**Solution**:

Root node (assuming split occurs):
```
        [40]
       /    \
[10,20,30]  [50,60,70,80,90]
```

Step 1: At root, 45 > 40, go to right child.
Step 2: At right child [50,60,70,80,90], search sequentially. 45 < 50, so 45 is not in this node.
Step 3: 45 is not found. (Or follow pointer before 50 to leaf level.)

Actually, in a B+ tree, all keys are at the leaf level. Internal nodes guide the search.

Root: [40]
- 45 > 40, go to right subtree

Internal node: [50, 70, 85]
- 45 < 50, go to leftmost leaf

Leaf node: [45, 46, 48] -> Found! (Assuming key 45 exists)

Number of disk accesses = tree height (levels) + 1 (leaf) = typically 2-4 for most databases.

### Problem 3: B+ Tree Insertion

Insert 25 into the B+ tree of order 4 from Problem 2.

**Solution**:

Initial root: [40]
Initial internal: [50, 70, 85] and internal for left subtree.

After inserting 25:
- 25 goes to leaf with [10, 20, 30]
- Leaf becomes [10, 20, 25, 30] (capacity 4, OK, no split)

No structural change needed.

Insert 35:
- Leaf [10, 20, 25, 30] becomes [10, 20, 25, 30, 35] (overflow!)
- Split into [10, 20, 25] and [30, 35]
- Promote 25 to parent
- Parent [25] merges into the left internal node

### Problem 4: Static Hashing

A hash table has 5 buckets. Hash function h(k) = k mod 5. Insert keys: 12, 25, 37, 18, 42, 30, 27.

**Solution**:

| Bucket | Keys |
|--------|------|
| 0 | 25, 30 |
| 1 |  |
| 2 | 12, 37, 27 |
| 3 | 18 |
| 4 | 42 |

- 12 mod 5 = 2 -> Bucket 2
- 25 mod 5 = 0 -> Bucket 0
- 37 mod 5 = 2 -> Bucket 2 (collision! chaining adds to bucket)
- 18 mod 5 = 3 -> Bucket 3
- 42 mod 5 = 4 -> Bucket 4
- 30 mod 5 = 0 -> Bucket 0 (collision)
- 27 mod 5 = 2 -> Bucket 2 (collision)

If using open addressing (linear probing), collisions are resolved by finding the next empty slot.

### Problem 5: B-Tree vs B+ Tree

Compare B-Tree and B+ Tree:

| Feature | B-Tree | B+ Tree |
|---------|--------|---------|
| Data pointers | In all nodes | Only in leaf nodes |
| Internal nodes | Store keys + data pointers | Store only keys (routing) |
| Leaf nodes | Same as internal nodes | Linked together for sequential access |
| Height | Typically shorter (data in internal nodes) | Slightly taller |
| Range queries | Slow (need inorder traversal) | Fast (follow leaf links) |
| Insert/delete | May need to move data pointers | Only leaf modification |
| Use case | Indexed sequential files | Database indexes (most RDBMS) |
