## Notes

### Topic Recap: Locking Protocols, Timestamp Schedulers

**Locking Protocols**:
- **Shared Lock (S)**: Read only. Multiple transactions can hold S-locks simultaneously.
- **Exclusive Lock (X)**: Read and write. Only one transaction can hold X-lock.
- **Two-Phase Locking (2PL)**: Growing phase (acquire locks) + Shrinking phase (release locks).
- **Strict 2PL**: Release locks only after commit/abort. Prevents cascading aborts.
- **Rigorous 2PL**: All locks released only after commit.

**Timestamp-Based Protocols**:
- Each transaction gets a unique timestamp (TS).
- Read timestamp (R-TS) and Write timestamp (W-TS) for each data item.
- **Thomas' Write Rule**: Ignore outdated writes.

### Problem 1: Lock Compatibility Matrix

Fill the lock compatibility matrix: Can T2 acquire a lock when T1 holds a lock?

| T1 holds | T2 requests S | T2 requests X |
|----------|---------------|---------------|
| S | Yes (shared read) | No (upgrade to X blocked) |
| X | No | No |

T1 holding S-lock: Other transactions can also acquire S-lock (multiple readers). X-lock request must wait.

T1 holding X-lock: No other transaction can acquire any lock. Must wait until T1 releases.

### Problem 2: 2PL Example

Schedule using 2PL:
```
T1: LOCK-S(A), READ(A), LOCK-X(B), READ(B), B=B+A, WRITE(B), UNLOCK(A), UNLOCK(B)
T2: LOCK-S(B), READ(B), LOCK-X(A), READ(A), A=A+B, WRITE(A), UNLOCK(B), UNLOCK(A)
```

Does this follow 2PL? Identify growing and shrinking phases.

**Solution**:

T1 growing phase: LOCK-S(A), LOCK-X(B) (acquiring locks)
T1 shrinking phase: UNLOCK(A), UNLOCK(B) (releasing locks)
T1 follows 2PL: All locks acquired before any release.

T2 growing phase: LOCK-S(B), LOCK-X(A)
T2 shrinking phase: UNLOCK(B), UNLOCK(A)
T2 follows 2PL.

Is this schedule possible? T1 locks A (S), T1 locks B (X). If T2 tries LOCK-S(B), it must wait because T1 holds X on B. Deadlock possible if T1 waits for T2 and vice versa.

### Problem 3: Deadlock Detection

Transaction T1 holds X(A), requests S(B).
Transaction T2 holds X(B), requests S(A).

Draw the wait-for graph and detect deadlock.

**Solution**:

Wait-for graph:
```
T1 -> T2 (T1 waits for B held by T2)
T2 -> T1 (T2 waits for A held by T1)
```

Cycle: T1 -> T2 -> T1. Deadlock detected.

**Resolution**: Choose a victim transaction to abort (e.g., T2, the younger one). Release its locks, allowing T1 to proceed.

### Problem 4: Timestamp Ordering

Transactions T1 (TS=100) and T2 (TS=200). Data items A and B, both initially R-TS=0, W-TS=0.

Schedule:
```
T1: READ(A)
T2: WRITE(A)
T1: READ(B)
T2: WRITE(B)
```

Apply basic timestamp ordering protocol.

**Solution**:

Step 1: T1:READ(A) at TS=100
- Check: W-TS(A)=0 <= 100. OK. Proceed.
- Set R-TS(A)=100.

Step 2: T2:WRITE(A) at TS=200
- Check: R-TS(A)=100 <= 200. OK. Proceed.
- Check: W-TS(A)=0 <= 200. OK. Proceed.
- Set W-TS(A)=200.

Step 3: T1:READ(B) at TS=100
- Check: W-TS(B)=0 <= 100. OK. Proceed.
- Set R-TS(B)=100.

Step 4: T2:WRITE(B) at TS=200
- Check: R-TS(B)=100 <= 200. OK.
- Check: W-TS(B)=0 <= 200. OK.
- Set W-TS(B)=200.

No conflicts. Both commit successfully.

### Problem 5: Timestamp with Thomas' Write Rule

T1 (TS=50): WRITE(A)
T2 (TS=100): WRITE(A)
T3 (TS=150): READ(A)

Schedule:
```
T1: WRITE(A)   -- W-TS(A)=50
T2: WRITE(A)   -- TS=100, W-TS(A)=50 <= 100, OK. W-TS(A)=100
T3: READ(A)    -- TS=150, W-TS(A)=100 <= 150, OK. R-TS(A)=150
```

Now a late write from T1 occurs:
T1: WRITE(A) -- but T1 has TS=50, W-TS(A)=100. TS < W-TS(A).

**Without Thomas' Write Rule**: T1's write is too old. Abort T1.

**With Thomas' Write Rule**: T1's write is ignored (outdated). T1 continues. This is safe because T1's write would be overwritten by T2's write anyway, and T3 reads the most recent value.
