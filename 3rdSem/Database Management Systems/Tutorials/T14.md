### Topic Recap: Authorization Models, Access Control Techniques, Encryption

**Encryption Types**:
- **Symmetric**: Same key for encrypt and decrypt (AES, DES). Fast.
- **Asymmetric**: Public/private key pair (RSA). Slower. Used for key exchange.

**Data Encryption Levels**:
- **At Rest**: Encrypted database files (TDE).
- **In Transit**: SSL/TLS for network communication.
- **In Use**: Homomorphic encryption (research).

**Auditing**: Tracking who accessed what and when.

### Problem 1: Symmetric vs Asymmetric Encryption

Compare symmetric and asymmetric encryption:

| Feature | Symmetric | Asymmetric |
|---------|-----------|------------|
| Key | Single shared key | Public + Private key pair |
| Speed | Fast | Slow (100-1000x slower) |
| Key exchange | Problematic (must share securely) | Easy (public key is public) |
| Algorithm | AES, DES, 3DES | RSA, ECC |
| Use case | Encrypting large data | Key exchange, digital signatures |
| Key length | 128-256 bits | 2048-4096 bits |

### Problem 2: Encryption at Rest

A hospital database stores patient records. Design an encryption strategy.

**Solution**:

1. **Transparent Data Encryption (TDE)**: Automatically encrypts data files on disk. Database engine encrypts/decrypts transparently. No application changes needed.

2. **Column-Level Encryption**: Encrypt sensitive columns (SSN, medical records) separately.

```sql
-- Create master key
CREATE MASTER KEY ENCRYPTION BY PASSWORD = 'StrongPassword123';

-- Create certificate
CREATE CERTIFICATE PatientCert WITH SUBJECT = 'Patient Data Protection';

-- Create symmetric key
CREATE SYMMETRIC KEY PatientKey
WITH ALGORITHM = AES_256
ENCRYPTION BY CERTIFICATE PatientCert;

-- Encrypt data on insert
OPEN SYMMETRIC KEY PatientKey DECRYPTION BY CERTIFICATE PatientCert;
INSERT INTO Patient VALUES (
    101,
    EncryptByKey(Key_GUID('PatientKey'), 'John Doe'),
    EncryptByKey(Key_GUID('PatientKey'), 'SSN-123-45-6789')
);
CLOSE SYMMETRIC KEY PatientKey;
```

3. **Application-Level Encryption**: Encrypt data in the application before sending to the database. Database sees only ciphertext.

### Problem 3: Digital Signatures

Explain how digital signatures provide integrity and non-repudiation.

**Signing**:
1. Sender computes hash of the message.
2. Sender encrypts the hash with their private key (this is the signature).
3. Sender sends message + signature.

**Verification**:
1. Receiver decrypts the signature using sender's public key, gets hash1.
2. Receiver computes hash of the received message, gets hash2.
3. If hash1 == hash2, integrity is verified and sender is authenticated.

**Provides**: Integrity (hash mismatch means tampering) and Non-repudiation (only sender's private key could have created the signature).

### Problem 4: Auditing in SQL Server

Set up auditing to track who modified the Employee table.

```sql
-- Create a server audit
CREATE SERVER AUDIT EmployeeAudit
TO FILE (FILEPATH = 'C:\AuditLogs\');

-- Create database audit specification
CREATE DATABASE AUDIT SPECIFICATION EmployeeAuditSpec
FOR SERVER AUDIT EmployeeAudit
ADD (SELECT, INSERT, UPDATE, DELETE ON Employee BY public);

-- Enable the audit
ALTER SERVER AUDIT EmployeeAudit WITH (STATE = ON);
ALTER DATABASE AUDIT SPECIFICATION EmployeeAuditSpec WITH (STATE = ON);
```

Audit log will record:
- Which user performed the operation
- When it was performed
- What type of operation (INSERT/UPDATE/DELETE)
- Old and new values (if configured)

### Problem 5: View-Based Access Control

Use views to restrict access to sensitive data.

```sql
-- Full employee table has salary column
CREATE TABLE Employee (
    EID INT PRIMARY KEY,
    Name VARCHAR(50),
    Department VARCHAR(20),
    Salary DECIMAL(10,2),
    SSN VARCHAR(11)
);

-- View for regular managers: no SSN
CREATE VIEW ManagerView AS
SELECT EID, Name, Department, Salary
FROM Employee;

-- View for HR: all columns
CREATE VIEW HRView AS
SELECT * FROM Employee;

-- View for employees: only own data
CREATE VIEW SelfView AS
SELECT Name, Department
FROM Employee
WHERE EID = SYSTEM_USER;

-- Grant access appropriately
GRANT SELECT ON ManagerView TO ManagerRole;
GRANT SELECT ON HRView TO HRRole;
GRANT SELECT ON SelfView TO Public;
```
