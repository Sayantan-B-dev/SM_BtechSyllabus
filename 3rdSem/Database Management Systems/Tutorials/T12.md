### Topic Recap: Database Recovery Techniques

**Recovery Concepts**:
- **Log**: Sequential record of all database modifications.
- **Log Record Types**: <T, START>, <T, X, old_value, new_value>, <T, COMMIT>, <T, ABORT>
- **Checkpoint**: Periodically flush log and dirty pages to disk.
- **Undo**: Restore old value (for aborted transactions).
- **Redo**: Apply new value (for committed transactions).
- **Deferred Update**: Write to log first, then database. No undo needed.
- **Immediate Update**: Write to database immediately, but log must be written first (Write-Ahead Logging).

### Problem 1: Write-Ahead Logging

Explain WAL with an example.

**Transaction T1**: A=100 -> A=200, B=200 -> B=150.

**WAL Rule**: Before a database change is written to disk, the corresponding log record must be written to stable storage.

**Log sequence**:
```
<T1, START>
<T1, A, 100, 200>  -- written to log BEFORE A is updated on disk
<T1, B, 200, 150>  -- written to log BEFORE B is updated on disk
<T1, COMMIT>
```

If crash occurs after log <T1, START> is written but before <T1, A, 100, 200>, no recovery action needed (no change was made).

If crash occurs after log <T1, A, 100, 200> but before <T1, B, 200, 150>, we know from the log that A was changed on disk. We UNDO: restore A=100.

### Problem 2: Recovery Using Log

Given this log after a crash:

```
<T1, START>
<T1, A, 10, 20>
<T2, START>
<T1, B, 30, 40>
<T2, C, 50, 60>
<T1, COMMIT>
<T2, COMMIT>
<CHECKPOINT>
<START CKPT(T1, T2)>
<END CKPT>
<T3, START>
<T3, D, 70, 80>
CRASH
```

Determine which transactions need UNDO and which need REDO.

**Solution**:

Transactions at checkpoint: T1 (committed before checkpoint), T2 (committed before checkpoint)

After checkpoint: T3 started but not committed.

Transactions that committed before checkpoint (in stable storage): T1, T2
- No recovery needed (already on disk).

Transaction active after checkpoint: T3
- T3 did not commit. Need UNDO (restore D=70).

If the checkpoint was not yet complete:
- <START CKPT(T1, T2)> was written but <END CKPT> was not.
- T1 and T2 committed, but their changes might not be on disk.
- T1: REDO (committed, changes might not be flushed)
- T2: REDO (committed, changes might not be flushed)
- T3: UNDO (uncommitted)

### Problem 3: ARIES Recovery Algorithm

Describe the three phases of ARIES recovery.

**Analysis Phase**: Read log from last checkpoint to determine:
- Dirty pages (pages modified but not yet written to disk)
- Transactions in progress (for UNDO)
- Transactions committed (for REDO)

**Redo Phase**: Reapply all changes from committed and incomplete transactions (starting from the smallest LSN in the dirty page table).

**Undo Phase**: Roll back all uncommitted transactions (in reverse order, using the log to restore old values).

### Problem 4: Deferred vs Immediate Update

Compare recovery approaches:

| Scenario | Deferred Update | Immediate Update |
|----------|----------------|-----------------|
| Log write | Before any DB write | Before DB write (WAL) |
| DB write | After commit | During transaction |
| Undo needed | No (abort just discards log) | Yes (restore old values) |
| Redo needed | Yes (reapply log after crash) | Yes |
| Crash during transaction | No undo needed, just discard | Undo needed |
| Performance | Less I/O during transaction | More I/O during transaction |

### Problem 5: Shadow Paging

Explain shadow paging recovery technique.

**Shadow Paging**: Maintain two page tables: current page table and shadow page table.

- **Current page table**: Points to current (modified) pages.
- **Shadow page table**: Points to original (unmodified) pages. Never modified.

**Transaction execution**:
1. Current page table is copied to shadow page table.
2. Modifications are made to the current page table (new pages allocated).
3. On commit: shadow page table is updated to point to new pages.
4. On abort: discard current page table, use shadow page table.

**Advantage**: No log needed (for undo). Fast recovery.
**Disadvantage**: Page fragmentation, more complex garbage collection.
